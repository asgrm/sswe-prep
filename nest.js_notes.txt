3. What is the purpose of @Res() in NestJS?
  It injects a library-specific response object (e.g., Express) to use native response handling methods.

4. What happens if both @Res() and the standard response approach are used simultaneously in a handler?
  The standard approach is automatically disabled for that route unless you enable it by setting @Res({ passthrough: true }).

5. How can you enable both @Res() and the standard response approach for the same route?
  Use @Res({ passthrough: true }).

10. How does the @Redirect() decorator work in NestJS?
  It redirects to a specified URL and status code but allows the handler to override the URL dynamically by returning a new one. Example

      return { url: 'https://docs.nestjs.com/v5/' };
    }
  }

32. How do you define a dynamic module in NestJS?
 By using a static method (forRoot() or forRootAsync()) that returns a DynamicModule object containing runtime-specific providers.
  Definition

    @Module({
      providers: [Connection],
      exports: [Connection],
    })
    export class DatabaseModule {
      static forRoot(entities = [], options?): DynamicModule {

        return {
          module: DatabaseModule,
          providers: providers,
          exports: providers,
        };
      }
    }

45. What is the role of the built-in exceptions layer in NestJS?

52 What is the purpose of exception filters in Nest.js?
  Exception filters give full control over the exception handling layer, enabling logging, dynamic JSON responses, or other custom logic.

53. What does the @Catch() decorator do?
  The @Catch() decorator binds metadata to an exception filter, specifying which exception types the filter handles.

54. How can an exception filter access the Request and Response objects?
  By using the ArgumentsHost object and methods like host.switchToHttp(), .getRequest<Request>(), and .getResponse<Response>().

55. Why is ArgumentsHost used in the catch() of exception filters method?
  ArgumentsHost abstracts the execution context, enabling exception filters to work across different contexts like HTTP, WebSockets, or Microservices

56. What method should exception filters implement?
  The catch(exception: T, host: ArgumentsHost) method, where T is the exception type.

57. Write the code for a custom exception filter for HttpException.
  @Catch(HttpException)
  export class HttpExceptionFilter implements ExceptionFilter {
    catch(exception: HttpException, host: ArgumentsHost) {
      const ctx = host.switchToHttp();
      const response = ctx.getResponse<Response>();
      const request = ctx.getRequest<Request>();
      const status = exception.getStatus();

      response
        .status(status)
        .json({
          statusCode: status,
          timestamp: new Date().toISOString(),
          path: request.url,
        });
    }
  }

64. Why is it usefull to register global filters within a module?
  It allows for dependency injection, which is not possible when registering filters outside the module context.
  Example:

  1) Dependency to be used:
  @Injectable()
  export class LoggingService {
    private readonly logger = new Logger(LoggingService.name);

    logException(exception: unknown) {
      this.logger.error(
        `Logging exception: ${JSON.stringify(exception)}`
      );
    }
  }

  2) Exception Filter

  @Catch(HttpException)
  export class HttpExceptionLoggerFilter implements ExceptionFilter {
    constructor(private readonly loggingService: LoggingService) {}

    catch(exception: HttpException, host: ArgumentsHost) {
      const ctx = host.switchToHttp();
      const response = ctx.getResponse<Response>();
      const request = ctx.getRequest<Request>();
      const status = exception.getStatus();

      this.loggingService.logException({
        statusCode: status,
        path: request.url,
        message: exception.message,
        timestamp: new Date().toISOString(),
      });

      response
        .status(status)
        .json({
          statusCode: status,
          timestamp: new Date().toISOString(),
          path: request.url,
        });
    }
  }

  3) enable global usage of the filter and inject the LoggingService in root module
  @Module({
    providers: [
      LoggingService,   // Ensure the LoggingService is injectable
      {
        provide: APP_FILTER,  // Register the global filter using dependency injection
        useClass: HttpExceptionLoggerFilter,
      },
    ],
  })
  export class AppModule {}

66. What are the two typical use cases for pipes in Nest.js?
  Transformation: Convert input data to the desired form (e.g., from string to integer).
  Validation: Evaluate input data and either pass it through unchanged if valid or throw an exception if invalid.

67. When do pipes operate in Nest.js during a request lifecycle?
  Pipes operate before a controller route handler is invoked, processing the arguments destined for the method. Nest interposes a pipe just before a method is invoked, and the pipe receives the arguments destined for the method and operates on them. Any transformation or validation operation takes place at that time, after which the route handler is invoked with any (potentially) transformed arguments.

68. What happens when a pipe throws an exception in Nest.js, and why is this considered a best practice for data validation?
  Pipes run inside the exceptions zone. This means that when a Pipe throws an exception it is handled by the exceptions layer (global exceptions filter and any exceptions filters that are applied to the current context). An exception is thrown in a Pipe, no controller method is subsequently executed. This gives you a best-practice technique for validating data coming into the application from external sources at the system boundary.

82. How to create custom validation pipe?
  it should extend `PipeTransform<T, R> ` interface.

  PipeTransform<T, R> is a generic interface that must be implemented by any pipe. The generic interface uses T to indicate the type of the input value, and R to indicate the return type of the transform() method.
  class should implement transform method, which take two params `value` and `metadata`
  metadata is of following type:
  export interface ArgumentMetadata {
    type: 'body' | 'query' | 'param' | 'custom';
    metatype?: Type<unknown>;
    data?: string;
  }

  `type` indicates whether it's a body, query, param   or a custom parameter

  simple example:
  @Injectable()
  export class ValidationPipe implements PipeTransform {
    transform(value: any, metadata: ArgumentMetadata) {
      return value;
    }
  }

83. What are the four scopes available for pipes in NestJS?
  Global-Scoped: Applied application-wide via app.useGlobalPipes().
  Controller-Scoped: Applied to an entire controller via @UsePipes() at the class level.
  Method-Scoped: Applied to a specific method (e.g., pipes passed in decorator arguments like @Post()).
  Parameter-Scoped: Applied to specific parameters in a method (e.g., @Body(new ValidationPipe())).

87. What is a guard in NestJS?
  A guard in NestJS is a class annotated with the @Injectable() decorator that implements the CanActivate interface.

  Guards have a single responsibility. They determine whether a given request will be handled by the route handler or not, depending on certain conditions (like permissions, roles, ACLs, etc.) present at run-time.

  This is often referred to as authorization.

  guard class shorl have method canActivate taking ExecutionContext (extengs ArgumentsHost) as single argument and returning boolean value (indicating whether to allow request).

88.  What are the key differences between middleware and guards in NestJS?

  Middleware: "Dumb" by nature, tied to the request/response cycle without an awareness of execution context. Does not know which route handler will be executed after next().

  Guards: Accesses the ExecutionContext instance, knowing which route handler will run next.

89. In which order related to pipes and middlewares guards are executed?
  Guards are executed after all middleware, but before any interceptor or pipe.

93. How do you create a custom authorization guard using roles in NestJS? Provide an example.

  Guard Implementation:

    @Injectable()
    export class RolesGuard implements CanActivate {
      constructor(private reflector: Reflector) {}

      canActivate(context: ExecutionContext): boolean {
        const roles = this.reflector.get('roles', context.getHandler()); // Retrieve roles metadata
        if (!roles) {
          return true; // If no roles are specified, allow access
        }
        const request = context.switchToHttp().getRequest();
        const user = request.user; // Assume the user object is attached to the request
        return matchRoles(roles, user.roles); // Check if user roles match required roles
      }
    }

  Roles Decorator:

    import { SetMetadata } from '@nestjs/common';
    export const Roles = (...roles: string[]) => SetMetadata('roles', roles);

  Using the Roles Guard and Decorator:

    @Post()
    @Roles(['admin']) // Attach required roles
    async create(@Body() createCatDto: CreateCatDto) {
      this.catsService.create(createCatDto); // Only 'admin' can access this endpoint
    }

96. Explain NestInterceptor interface.
  method intercept takes 2 params: ExecutionContext instance and CallHandler

  The CallHandler interface implements the handle() method, which you can use to invoke the route handler method at some point in your interceptor.

  If you don't call the handle() method in your implementation of the intercept() method, the route handler method won't be executed at all. Once handle() is called (and its Observable has been returned), the method's handler will be triggered.

  handle() method returns an Observable, as a result, you may implement custom logic both before and after the execution of the final route handler.
  Once the response stream is received via the Observable, additional operations can be performed on the stream, and a final result returned to the caller.

97. Can Interceptors inject dependencies?
  Yes as well as controllers, providers, guards, and so on, they can inject dependencies through their constructor.

98. Provide example of Timeout Interceptor.
  @Injectable()
  export class TimeoutInterceptor implements NestInterceptor {
    constructor(private readonly timeoutMs: number = 5000) {} // Default timeout is 5000ms

    intercept(context: ExecutionContext, next: CallHandler): Observable<any> {
      return next.handle().pipe(
        timeout(this.timeoutMs), // Apply timeout
        catchError((err) => {
          // Throw a specific exception for timeout events
          if (err.name === 'TimeoutError') {
            throw new RequestTimeoutException('Request timed out');
          }
          throw err; // Re-throw other errors
        }),
      );
    }
  }

99. Provide example of Logging interceptor.
  @Injectable()
  export class LoggingInterceptor implements NestInterceptor {
    intercept(context: ExecutionContext, next: CallHandler): Observable<any> {

111. How does the Nest IoC container handle dependency injection, and what is the role of the dependency graph?

  When instantiating a controller (e.g., `CatsController`), the Nest IoC container looks for dependencies (e.g., `CatsService`).
  For each dependency, it performs a lookup on the token, retrieves the class, and resolves it.
  For SINGLETON scope (default), it creates an instance of the dependency, caches it, and reuses it for subsequent requests. This process is transitive, meaning if a dependency (e.g., `CatsService`) has its own dependencies, they are resolved first.

  This "dependency graph" ensures correct resolution order ("bottom-up") during application bootstrapping, simplifying complex dependency management for developers.

117. How can you use the useClass syntax for dynamic class-based provider resolution in NestJS?
  The useClass syntax allows dynamically choosing a class implementation for a token during runtime, based on conditions like environment variables.

    useClass:

  })

118. How does the useFactory syntax work for creating dynamic providers in NestJS?
  The useFactory syntax allows for generating providers dynamically by using a factory function. The factory function can optionally depend on other providers.

    const connectionProvider = {
      provide: 'CONNECTION',
      useFactory: (optionsProvider: MyOptionsProvider, optionalProvider?: string) => {
        const options = optionsProvider.get();
        return new DatabaseConnection(options);
      },
      inject: [MyOptionsProvider, { token: 'SomeOptionalProvider', optional: true }],
      //       \______________/             \__________________/
      //        This provider                The provider with this token
      //        is mandatory.                can resolve to `undefined`.
    };

    @Module({
      providers: [
        connectionProvider,
        MyOptionsProvider, // class-based provider
        // { provide: 'SomeOptionalProvider', useValue: 'anything' },
      ],
    })
    export class AppModule {}


  Key Details:

    Factory Function: The factory function creates the provider dynamically, and its return value is used as the provider.
      Example: new DatabaseConnection(options) provides the final object.

    Arguments: The factory function can accept arguments, which are instances of other providers resolved by Nest.

    Inject Property: The inject array specifies the tokens of dependencies to be resolved and passed as arguments to the factory function (in the same order).

      Example: MyOptionsProvider for mandatory dependency.
      Optional providers (like 'SomeOptionalProvider') are marked with { optional: true } and can resolve to undefined.

    Flexibility: This method is highly customizable and allows dynamic creation of providers, incorporating both mandatory and optional dependencies.

122. What is a dynamic module in NestJS, and how can it be used with a static register()/forRoot() method?
  A dynamic module in NestJS is a module created at runtime, with all the same properties as a static module, plus one required property called module.

  Dynamic modules must return an object with all typical module properties (e.g., providers, imports, exports) and an additional module property.

  The @Module() decorator's imports property can take a dynamic module returned from a static method (e.g., ConfigModule.register()).
    Example (Dynamic Module registeration):

      import { Module } from '@nestjs/common';
      import { AppController } from './app.controller';
      import { AppService } from './app.service';
      import { ConfigModule } from './config/config.module';

      @Module({
        imports: [ConfigModule.register({ folder: './config' })],  // Using ConfigModule's static method
        controllers: [AppController],
        providers: [AppService],
      })
      export class AppModule {}

    Example (Dynamic Module):

      @Module({})
      export class ConfigModule {
        static register(options: Record<string, any>): DynamicModule {
          return {
            module: ConfigModule,
            providers: [
              {
                provide: 'CONFIG_OPTIONS',
                useValue: options,
              },
              ConfigService,
            ],
            exports: [ConfigService],
          };
        }
      }

    Example (downstream service):

      @Injectable()
      export class ConfigService {
        private readonly envConfig: EnvConfig;

        constructor(@Inject('CONFIG_OPTIONS') private options: Record<string, any>) {
          const filePath = `${process.env.NODE_ENV || 'development'}.env`;
          const envFile = path.resolve(__dirname, '../../', options.folder, filePath);
          this.envConfig = dotenv.parse(fs.readFileSync(envFile));
        }

        get(key: string): string {
          return this.envConfig[key];
        }
      }

  A dynamic module can encapsulate logic and dependencies, making it reusable with varying configurations in different parts of the application.

123. What is the difference between `register`, `forRoot`, and `forFeature` methods in NestJS modules?

  register: Used to configure a dynamic module with a specific configuration that is local to the calling module.

  forRoot: Used to configure a dynamic module once with a shared configuration that is reused globally across the application. Typically used for modules that provide global services or resources (e.g., database connections, GraphQL, etc.).

  forFeature: Used to build on the shared configuration from forRoot but adjust specific settings for the calling module's individual needs. For example typeorm for specific repo.

  registerAsync/forRootAsync/forFeatureAsync - async version

  *it's a Community guideline, not a rule.

124. What is the ConfigurableModuleBuilder in NestJS, and how does it simplify the creation of configurable, dynamic modules?

      @Injectable()

126. What are Provider scopes in Nest.js and how to define it?
  DEFAULT	- A single instance of the provider is shared across the entire application. The instance lifetime is tied directly to the application lifecycle.
  REQUEST - A new instance of the provider is created exclusively for each incoming request. The instance is garbage-collected after the request has completed processing.
  TRANSIENT - Transient providers are not shared across consumers. Each consumer that injects a transient provider will receive a new, dedicated instance.

  DEFAULT scope is set already unless different scope chosen.

  1) specify scope at Injectable decorator
    @Injectable({ scope: Scope.REQUEST })

  2) OR (for custom provider), specify in provider options

    {
      provide: 'CACHE_MANAGER',
      useClass: CacheManager,
      scope: Scope.TRANSIENT,
    }

129. How does scope hierarchy work with request-scoped and transient-scoped providers in NestJS?
  The REQUEST scope bubbles up the injection chain. A controller that depends on a request-scoped provider will, itself, be request-scoped.

    Example: dependency graph: CatsController <- CatsService <- CatsRepository. If CatsService is request-scoped (and the others are default singletons), the CatsController will become request-scoped as it is dependent on the injected service. The CatsRepository, which is not dependent, would remain singleton-scoped.

  Transient-scoped dependencies don't follow that pattern. If a singleton-scoped service injects a transient provider, , it will receive a fresh instance of it but service stay singleton-scoped.

132. What are durable providers in NestJS, and when should they be used?

  Durable providers optimize multi-tenant applications by creating persistent DI sub-trees per tenant instead of per request. This avoids unnecessary instantiation for tenant-related providers across repeated requests.

  Use them if classification can aggregate requests per tenant and does not rely on unique request attributes (e.g., UUID).

  HOWEVER, it isn't ideal for applications with a large number of tenants.

133. How can you define a ContextId strategy to aggregate requests by tenant for durable providers?

  You should use the ContextIdStrategy interface to group requests by tenant:

    import { ContextId, ContextIdFactory, ContextIdStrategy } from '@nestjs/core';
    import { Request } from 'express';

    const tenants = new Map<string, ContextId>();

    export class AggregateByTenantContextIdStrategy implements ContextIdStrategy {
      attach(contextId: ContextId, request: Request) {
        const tenantId = request.headers['x-tenant-id'] as string;
        let tenantSubTreeId: ContextId;

        if (tenants.has(tenantId)) {
          tenantSubTreeId = tenants.get(tenantId);
        } else {
          tenantSubTreeId = ContextIdFactory.create();
          tenants.set(tenantId, tenantSubTreeId);
        }

        return (info: HostComponentInfo) =>
          info.isTreeDurable ? tenantSubTreeId : contextId;
      }
    }

      }
    }
  Here, the CatsService is marked durable, meaning it will persist for a specific tenant's context instead of being destroyed after each request.

  2. Create a Tenant-Specific BucketService
  The BucketService determines the correct bucket (e.g., Bucket 1 or Bucket 2) for each tenant.

    import { Injectable, Scope } from '@nestjs/common';

    @Injectable({ scope: Scope.REQUEST, durable: true })
    export class BucketService {
      constructor(private readonly request: { tenantId: string }) {}

      getBucketName(): string {
        const tenantId = this.request.tenantId;
        // Map tenantId to a bucket name
        return tenantId === 'tenant-A' ? 'Bucket 1' : 'Bucket 2';
      }
    }

  Here, the BucketService accesses the tenant-specific tenantId injected from the durable context.

  3. Define the ContextId Strategy
  Aggregate requests based on the x-tenant-id header. This strategy ensures that all components sharing the same tenant context (e.g., CatsService and BucketService) belong to the same DI subtree.


    import {
      HostComponentInfo,
      ContextId,
      ContextIdFactory,
      ContextIdStrategy,
    } from '@nestjs/core';
    import { Request } from 'express';

    const tenants = new Map<string, ContextId>();

    export class AggregateByTenantContextIdStrategy implements ContextIdStrategy {
      attach(contextId: ContextId, request: Request) {
        const tenantId = request.headers['x-tenant-id'] as string;
        let tenantSubTreeId: ContextId;

        if (tenants.has(tenantId)) {
          tenantSubTreeId = tenants.get(tenantId);
        } else {
          tenantSubTreeId = ContextIdFactory.create();
          tenants.set(tenantId, tenantSubTreeId);
        }

        return {
          resolve: (info: HostComponentInfo) =>
            info.isTreeDurable ? tenantSubTreeId : contextId,
          payload: { tenantId }, // Add tenantId to the payload
        };
      }
    }

  4. Register the ContextId Strategy Globally
  Register the strategy in main.ts. This ensures that the strategy is globally applied across all request-scoped components.


    import { ContextIdFactory } from '@nestjs/core';
    import { AggregateByTenantContextIdStrategy } from './aggregate-by-tenant-context-id.strategy';

    async function bootstrap() {
      const app = await NestFactory.create(AppModule);

      // Apply tenant-based strategy
      ContextIdFactory.apply(new AggregateByTenantContextIdStrategy());

      await app.listen(3000);
    }
    bootstrap();

  5. Inject the Tenant Information via Payload
  When tenantId is added to the payload, it can be injected into request-scoped providers

    @Injectable({ scope: Scope.REQUEST, durable: true }) // Proper scope and durability settings
    export class BucketService {

      }
    }
  6. Create a Controller to Handle Requests
  The controller will route user requests to the appropriate tenant-specific CatsService.


    import { Controller, Get, Req } from '@nestjs/common';
    import { CatsService } from './cats.service';

    @Controller('cats')
    export class CatsController {
      constructor(private readonly catsService: CatsService) {}

      @Get('photos')

157. What are the lifecycle hooks in NestJS and when are they triggered?

  NestJS lifecycle hooks are triggered during application initialization and shutdown. They allow developers to perform specific tasks at key points in the lifecycle:

  Initialization Hooks:

    - onModuleInit(): Called when the host module's dependencies are resolved.
    - onApplicationBootstrap(): Called after all modules are initialized but before the application starts listening for connections. (Triggered by app.init() or app.listen().)

  Shutdown Hooks ( To enable shutdown hooks on system signals (like SIGTERM), explicitly call enableShutdownHooks()):

    - onModuleDestroy(): Called after a termination signal (e.g., SIGTERM) is received.
    - beforeApplicationShutdown(): Called after onModuleDestroy() completes, allowing time to close connections via app.close().
    - onApplicationShutdown(): Called after all connections are closed (i.e., when app.close() resolves).

158. Are lifecycle hooks (e.g., onModuleInit, onApplicationShutdown) triggered for request-scoped classes in NestJS?
  No, lifecycle hooks are not triggered for request-scoped classes in NestJS, since they are  tied to the lifespan of a single request. Meaning they are automatically garbage-collected after the response is sent

159. What determines the execution order of onModuleInit() and onApplicationBootstrap() lifecycle hooks in NestJS?

  The execution order of onModuleInit() and onApplicationBootstrap() depends on:

  1) The order of module imports:
    Hooks will be executed in the same order as modules are defined in the imports array of the root module or other relevant modules.

  2) Awaiting the previous hook:
    NestJS ensures that each lifecycle hook in an earlier module finishes execution before moving to the next one in the import order.

162. How can you enable application shutdown hooks in NestJS, and when are they triggered?

  Shutdown hooks are disabled by default. To enable them, call app.enableShutdownHooks() during app bootstrap.

  Example:
    import { NestFactory } from '@nestjs/core';
    import { AppModule } from './app.module';

    async function bootstrap() {
      const app = await NestFactory.create(AppModule);

      // Enable shutdown hooks
      app.enableShutdownHooks();

      await app.listen(process.env.PORT ?? 3000);
    }
    bootstrap();

  These hooks are triggered during shutdown in response to:

    Explicit call to app.close()
    System signals: Such as SIGTERM, often used in platforms like Kubernetes or Heroku to manage container lifecycles.

163. What platform limitations exist for application shutdown hooks in NestJS on Windows?

  Platform Limitations on Windows:
    SIGINT: Fully supported (e.g., triggered via Ctrl+C).
    SIGBREAK: Supported (e.g., triggered via Ctrl + Break).
    SIGHUP: Partially supported.

    SIGTERM is not supported on Windows because killing a process in the Task Manager is unconditional — there is no way for an application to detect or prevent it.

Side Note: `enableShutdownHooks` consumes memory by starting listeners.

164. What does calling app.close() do in a NestJS application, and does it terminate the Node.js process?
  Calling app.close() does not terminate the Node.js process.
  It triggers the following shutdown hooks: onModuleDestroy() -> onApplicationShutdown()

  If the application has active intervals, long-running background tasks, or other asynchronous operations, those tasks will prevent the Node.js process from terminating automatically.

207. What are discriminators in MongoDB (Mongoose)? How do they function, and give an example of their usage?

211. What is a virtual in Mongoose, and how is it used? Provide an example.
  A virtual is a computed property in Mongoose that exists in a document but is not stored in the database.
  It can be used for derived values (e.g., concatenating fields) or calculations based on data in the document.

  Example:
    import { Prop, Virtual } from '@nestjs/mongoose';

    class Person {
      @Prop()
      firstName: string;

      @Prop()
      lastName: string;

      @Virtual({
        get: function (this: Person) {
          return `${this.firstName} ${this.lastName}`;
        },
      })
      fullName: string; // Computed property
    }

  Usage: The fullName virtual dynamically combines firstName and lastName but is not saved to MongoDB.

258. What are the advantages of running BullMQ job handlers in separate (forked) processes, and how do you configure it

  Sandboxed execution - Process crashes don't affect the main worker
  Non-blocking code - Can run blocking code without stalling other jobs

289. What are the three essential components needed for encryption?

  Algorithm: The cipher method (e.g., 'aes-256-ctr')
  Key: Secret password that controls encryption/decryption (32 bytes for AES-256)
  IV (Initialization Vector): Random data that makes encryption unpredictable (16 bytes)

  example:

    const iv = randomBytes(16); // Random 16 bytes
    const key = (await promisify(scrypt)(password, 'salt', 32)) as Buffer; // 32-byte key
    const cipher = createCipheriv('aes-256-ctr', key, iv);

  Analogy: Key = house key, IV = changing the lock position each time, Algorithm = type of lock

291. What's the main challenge with encryption IVs in distributed systems (multiple pods/servers)?

  Each pod generates different random IVs during encryption:

    Pod A encrypts data → creates random IV₁
    Pod B tries to decrypt → doesn't know IV₁ → decryption fails

  Solutions
    1) Prepend IV to encrypted data, store together as single blob.

      Encryption:
        const iv = randomBytes(16);
        const cipher = createCipheriv('aes-256-ctr', key, iv);
        const encrypted = Buffer.concat([cipher.update(text), cipher.final()]);

        // Store IV + encrypted data together
        const encryptedWithIV = Buffer.concat([iv, encrypted]);
        // Save encryptedWithIV to database

      Decryption:
        const storedData = getFromDatabase(); // IV + encrypted data
        const iv = storedData.subarray(0, 16);        // First 16 bytes = IV
        const encrypted = storedData.subarray(16);     // Rest = encrypted data

        const decipher = createDecipheriv('aes-256-ctr', key, iv);
        const decrypted = Buffer.concat([decipher.update(encrypted), decipher.final()]);

    2) use external secret services (AWS KMS, HashiCorp Vault, Azure Key Vault, Google Secret Manager)
      Flow:
        Pod A encrypts → stores key/IV pair in service with unique ID
        Pod A stores encrypted data + ID in database
        Pod B retrieves encrypted data + ID from database
        Pod B fetches key/IV from service using ID
        Pod B decrypts successfully

    3) store IV separated
      3.1) Store IV as separate field alongside encrypted data
        OR
      3.2)Use Redis with key pattern like encryption:${id}:iv and encryption:${id}:data

306.What are GraphQL resolvers and what do they do?
  Resolvers provide instructions for turning GraphQL operations (queries, mutations, subscriptions) into data.

  Key characteristics:

    Can return data synchronously or as a promise

307. How do you create a resolver class in NestJS code-first approach?

  Use the @Resolver() decorator on a class with resolver methods:

    @Resolver(() => Author)
    export class AuthorsResolver {
      constructor(
        private authorsService: AuthorsService,
        private postsService: PostsService,
      ) {}

      @Query(() => Author)
      async author(@Args('id', { type: () => Int }) id: number) {
        return this.authorsService.findOneById(id);
      }

      @ResolveField()
      async posts(@Parent() author: Author) {
        const { id } = author;
        return this.postsService.findAll({ authorId: id });
      }
    }

308. What's the difference between @Query() and @ResolveField() resolvers?

  @Query() resolvers:

    Top-level entry points to your GraphQL API
    Handle root queries like author(id: 1)
    Define what data can be requested

  @ResolveField() resolvers:

    Resolve individual fields of an object type
    Called when a specific field is requested
    Used for nested data or computed fields

  // Query resolver - entry point
  @Query(() => Author)
  async author(@Args('id') id: number) {
    return this.authorsService.findOneById(id);
  }

  // Field resolver - resolves Author.posts field
  @ResolveField()
  async posts(@Parent() author: Author) {
    return this.postsService.findAll({ authorId: author.id });
  }

309. Why and when do you need to pass a type to @Resolver() decorator?

  The @Resolver(() => Type) parameter is optional but required when you have field resolvers.

  Specifies the parent object type for field resolvers
  Enables @Parent() decorator to access the parent object
  Required when class contains @ResolveField() methods

  @Resolver(() => Author)  // Author is parent type
  export class AuthorsResolver {
    @ResolveField()  // Resolves Author.posts field
    async posts(@Parent() author: Author) {
      // @Parent() gives access to the Author object
      return this.postsService.findAll({ authorId: author.id });
    }
  }

Side note: Field resolvers are called automatically by GraphQL when their field is requested. You never call them manually - GraphQL handles the orchestration and passes the parent object via @Parent().

310. How do you decouple method names from GraphQL query names in NestJS resolvers?

  By default, GraphQL uses the method name as the query name:
    @Query(() => Author)
    async author(@Args('id') id: number) {
      // Creates: author(id: Int!): Author
    }

  To decouple, pass the desired GraphQL `name` to the decorator:
    @Query(() => Author, { name: 'author' })
    async getAuthor(@Args('id') id: number) {
      // Method: getAuthor()
      // GraphQL query: author(id: Int!): Author
    }

    @ResolveField('posts', () => [Post]) // name as first argument here
    async getPosts(@Parent() author: Author) {
      // Method: getPosts()
      // GraphQL field: posts
    }

  Benefits:

    Use descriptive method names (e.g., getAuthor())
    Keep clean GraphQL API names (e.g., author)
    Better code organization and readability

318. How do you publish events for GraphQL subscriptions in NestJS, and what's the critical requirement for the payload shape?

  @Mutation(() => Comment)
  async addComment(
    @Args('postId', { type: () => Int }) postId: number,
    @Args('comment') comment: CommentInput,
  ) {
    const newComment = this.commentsService.addComment({ id: postId, comment });

    // Publish event with shaped payload
    pubSub.publish('commentAdded', { commentAdded: newComment });

    return newComment;
  }

  *Critical requirement - Payload shape must match subscription:
    type Subscription {
      commentAdded(): Comment!  # ← Top-level property name
    }

    // Correct - matches subscription shape
      pubSub.publish('commentAdded', { commentAdded: newComment });

    // Wrong - shape mismatch will cause validation failure
      pubSub.publish('commentAdded', newComment);

339. What is the WebSocketAdapter interface in NestJS and how do you create a custom Redis adapter for load-balanced instances?

  WebSocketAdapter Interface Methods:

  - create - Creates socket instance
  - bindClientConnect - Binds client connection event
  - bindClientDisconnect - Binds client disconnection event (optional)
  - bindMessageHandlers - Binds incoming messages to handlers
  - close - Terminates server instance



  export class RedisIoAdapter extends IoAdapter {
    private adapterConstructor: ReturnType<typeof createAdapter>;

    async connectToRedis(): Promise<void> {
      const pubClient = createClient({ url: `redis://localhost:6379` });
      const subClient = pubClient.duplicate();
      await Promise.all([pubClient.connect(), subClient.connect()]);

    createIOServer(port: number, options?: ServerOptions): any {
      const server = super.createIOServer(port, options);
      server.adapter(this.adapterConstructor);
      return server;
    }
  }

351. What is the Kafka microservice message pattern in Nest.js and what key requirements must be met for proper operation?
  The Kafka microservice message pattern uses two topics (request and reply channels) where:

    - ClientKafkaProxy#send() sends messages with correlation ID, reply topic, and reply partition
    - The ClientKafkaProxy instance must be subscribed to reply topic and assigned to at least one partition before sending
    - Critical requirement: You need at least one reply topic partition for every Nest application running

  Partition Assignment Problem: When new consumers join, normal round-robin partitioning can reassign existing consumers to different partitions, causing them to lose response messages from pre-rebalance requests.

  Solution: Nest uses a custom partitioner that sorts consumers by high-resolution timestamps (process.hrtime()) set at application launch, preventing message loss during rebalancing.

352. When and how should you use the heartbeat callback in Kafka message handlers?

  When to use: For handlers with slow processing time that might exceed the session timeout.

  Purpose: Prevents Kafka from timing out the consumer session during long-running message processing.

  How to use:
  - Get heartbeat function from KafkaContext using context.getHeartbeat()
  - Call await heartbeat() between slow processing steps

  Example pattern:
    @MessagePattern('hero.kill.dragon')
    async killDragon(@Payload() message, @Ctx() context: KafkaContext) {
      const heartbeat = context.getHeartbeat();
      await doWorkPart1();
      await heartbeat(); // Prevent session timeout
      await doWorkPart2();
    }
